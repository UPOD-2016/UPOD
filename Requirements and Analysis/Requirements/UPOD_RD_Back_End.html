1. Introduction

1.1 Scope

UPOD's back end consists of three parts: a server, a database, and an application. The database stores data (i.e. categories, subcategories, and articles) in required data structures, and provides users access to the stored data through the use of search bar.

1.2 Definitions, Acronyms, and Abbreviations

UPOD

  • Undergraduate Physics Online Database

    • Moderator

  • A moderator can add/delete/edit categories, sub-categories, and articles.

    • User

  • A user can view categories/sub-categories/articles, search for categories/sub-categories/articles, and interact with diagrams.

    • Category (and Sub-category)

  • A set of articles that fall into a definable block of information.

    • Article

  • Viewable page containing information, formulas, and diagrams relating to a particular physics concept.

    • Diagram

  • Illustration of physics concepts, which can be static or dynamic, and also supports user interaction.

    • SVG (Scalable Vector Diagram)

  • Image format for diagrams with support for animation and interaction.

    • 2. Overall Description

    2.1 Product Perspective

    UPOD's back end implementation needs the collaboration of both front end and back end teams, as the functionalities provided by the database and scripting rely on the front-end design.

    2.1.1 System Interfaces

    UPOD operates on modern and standard browsers, including:

  • Google Chrome - version 51
  • Mozilla Firefox - version 47
  • Internet Explorer - version 11
  • Microsoft Edge - version 38
  • Safari - version 9.1

    • 2.1.2 Software Interfaces

    Database

    In the database of UPOD, the types of data stored include categories, sub-categories, and articles.

  • Ensure information (i.e. categories, sub-categories, and articles) is always stored only once.
  • Transform any data entered by moderators into required data structures.
  • Enable multiple users to access the database simultaneously.
  • Accept different moderators' requests through different network environments.

    • 2.1.3 Communications Interfaces

  • UPOD will remain responsive across internet service providers.

    • 2.2 Constraints

  • UPOD's backend must be implemented onto the site before the end of the 2016 spring term (August 10, 2016).
  • The budget of UPOD is $0, so any software must be free to use.

    • 3. Specific Requirements

    3.1 Functions

    The following is a list of use cases that are available for actors:

    Login

  • In the login page, a moderator needs to enter their username and password to authenticate their identity.

    • Logout

  • To terminate their section, a moderator can click the Logout option.

    • Add/Delete/Edit Categories (or Sub-Categories)

  • A moderator can add, delete, or edit categories/sub-categories in UPOD.

    • Add/Delete/Modify Articles

  • A moderator can add, delete, or edit articles in UPOD.

    • View Categories/Sub-Categories

  • A user can view categories/sub-categories that are available in UPOD.

    • View Articles

  • A user can view articles that are available in UPOD.

    • Search for Categories/Sub-Categories/Articles

  • By inputting keywords in the search bar, a user can view the related categories/sub-categories/articles in the search results page.

    • Interact with Diagrams

  • A user can interact with diagrams that are in SVG format.

    • 3.2 Software System Attributes

    3.2.1 Reliability

  • Database provides up-to-date, and accurate information.
  • No duplicate information is stored in the database.
  • The search engine should provide a list of articles that contains the key words or are related to the key words.

    • 3.2.2 Security

  • Account passwords of administrators and moderators accounts should be stored securely.
  • Only moderators can add, delete, or edit categories/sub-categories/articles.

    • 3.2.3 Maintainability

  • Additional features may be added to the website in the future, so the whole software should be well documented and easy to maintain.

    • 3.3 Categories Information

    3.3.1 Categories Notes

    • Categories are top level in terms of hierarchy.
    • Each sub-category will belong to ONE category (foreign key stored in the sub-category table).
    • Each sub-category will contain a set of articles that relate to that topic.
    • Each article can belong to ONE OR MORE sub-category.
    • Each article MUST belong to at least one sub-category.
    • Articles can only belong to sub-categories but not categories (however the belonging sub-category would belong to a parent category).

    3.3.2 Categories To-Do List

    1. Implement database schema for categories (category table, sub-category table (contains foreign key of category it belongs to), sub-category-to-article table)
    2. Create category MODEL
    3. Update category MODEL
    4. Remove category MODEL (delete category and all sub-categories (call delete sub-category function listed below for each sub-category in the category to be deleted) within this category)
    5. Create sub-category MODEL
    6. Update sub-category MODEL
    7. Remove sub-category MODEL (delete sub-category and all articles within this category. Need to check if article belongs to other sub-categories before deleting)
    8. Get all categories MODEL
    9. Get all categories with sub-categories and articles MODEL
    10. Get all sub-categories with articles in specific category MODEL
    11. Get all articles in specific sub-category MODEL
    12. Add article to sub-category MODEL (create new record in sub-category-to-article table)
    13. Update article to sub-category MODEL
    14. Remove article to sub-category MODEL
    15. Create VIEW/CONTROLLER to have a page to display list of all categories and sub-categories
    16. Create VIEW/CONTROLLER to have a page to display list of all articles in a specific category
    17. Integrate add/edit/delete category/sub-category functionality into create/edit article pages
